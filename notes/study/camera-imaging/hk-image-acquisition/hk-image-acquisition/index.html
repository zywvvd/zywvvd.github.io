海康相机取图方式整理
Yiwei Zhang 2026年7月9日 晚上
4.6k 字 39 分钟 0 次   0 人   0 条

本文最后更新于：2026年7月9日 晚上

一台海康网络相机，把图像拿出来的通路不止一条。本文按协议栈把取图方式分成 SDK、HTTP（ISAPI）、流协议、封装层四级，记录各自的端口、认证与可操作接口，并附 Python 实测脚本与输出。

分层模型

①②③ 是彼此正交的数据出口；④ 不提供新出口，只把底下三层的接口封装成更便利的工具。判断一个工具属于哪一层，看它实际收发数据走的是 SDK 私有协议、HTTP、还是 RTSP。

SDK 层（HCNetSDK）

能力最全、延迟低，但耦合海康且依赖 .so 环境。Python 无官方绑定，须经 ctypes 手写结构体映射。按返回形态分四条路径：

SDK 函数 返回 形态
NET_DVR_CaptureJPEGPicture 设备端编码的全分辨率 JPEG 单张 JPEG
NET_DVR_CapturePicture 从已开预览句柄抓一帧 BMP
NET_DVR_RealPlay_V40 + NET_DVR_SaveRealData / 回调 实时码流 H.264 / 265 码流
NET_DVR_GetFileByName / GetFileByTime 设备 SD 卡 / NVR 历史录像 录像文件

环境配置（Linux）

HCNetSDK 在 Linux 下的失败几乎都来自库加载：

项 说明
LD_LIBRARY_PATH libhcnetsdk.so 的 NEEDED 库（libhpr.so、libHCCore.so 等）无 RPATH，必须把 <sdk>/lib 加进搜索路径，否则 CDLL 直接报 cannot open shared object file。
组件库路径 lib/HCNetSDKCom/ 下组件库由主库运行时 dlopen，用 NET_DVR_SetSDKInitCfg(2, SDK_PATH) 告知 lib/ 目录。
OpenSSL 版本 SDK 自带 libcrypto.so.1.1 / libssl.so.1.1（OpenSSL 1.1）。系统库若是 OpenSSL 3.x，不能混用——用 SetSDKInitCfg(3/4) 显式指向 SDK 自带的那两个，避免符号冲突。
登录模式 byLoginMode=0（Private）走 8000 私有协议端口；1 走 ISAPI；2 自适应。
glibc SDK 编译于较老 glibc，新发行版一般向下兼容；反向则可能 GLIBC_XXX not found。

JPEG 质量档与体积

wPicQuality 决定设备端 JPEG 的量化表，进而决定体积与质量。同场景一次登录连抓 q0/q1/q2 + ISAPI：

来源 wPicQuality 体积 亮度量化表均值
SDK 0（最高） 389 KB 23.1
SDK 1（较好） 231 KB 46.1
SDK 2（一般） 180 KB 72.1
ISAPI /picture 设备默认 180 KB 72.1

为何「亮度量化表均值」代表质量：JPEG 有损只发生在一步——8×8 块 DCT 后，把 64 个系数除以量化表对应步长、再取整。步长小→系数几乎不变→细节保留（质量高、文件大）；步长大→系数多被除到 0→高频细节丢弃（质量低、文件小）。64 个步长的均值即整体量化激进程度，与质量反相关。它是代理量、非绝对 Q——真质量要 PSNR/SSIM 对未压缩原图，设备内拿不到，故量化表是能拿到的最好代理。

几个推论：ISAPI /picture 的质量与 SDK q2 逐位一致（均值均 72.1），即 ISAPI 走通道默认「一般」档；体积比随场景浮动（同为 q0、均值 23.1，细节多的场景 1.08 MB、平缓场景 389 KB，而 q2 恒 ~180 KB）——量化表只决定「质量天花板」，画面高频多则同质量下文件更大。判断质量别用 identify -format %q（海康自定义量化表，全报 16），直接读 PIL.Image.open(f).quantization[0] 取平均。

HTTP 层（ISAPI）

ISAPI 是基于 HTTP 的 REST 接口，开放、跨语言，无需 SDK。浏览器 Web 页与多数 Python 包抓图，底层即此处。抓图端点：

能力 端点
JPEG 抓图 /ISAPI/Streaming/channels/N/picture
MJPEG 预览 /ISAPI/Streaming/channels/102/httpPreview
录像检索 / 下载 /ISAPI/ContentMgmt/search、/downloadURI
事件订阅 /ISAPI/Event/notification/alertStream
能力探测 /ISAPI/System/capabilities

ISAPI 的 JPEG 质量本层不可调：实测 /picture?quality=0、?JPEGQuality=0、?snapShotQuality=0、?resolution=0 均不改变量化表（均值恒 72.1）；通道 capability 只暴露视频流质量（fixedQuality），无抓图 JPEG 质量档。即 ISAPI 抓图锁死在设备默认（= SDK q2），要拿高质量图只能走 SDK。

主码流 H.264 Main @ 2560×1440 25fps。注意 RTSP 给的是编码码流，帧由主机端解码得到——rtsp_frame.jpg 是 ffmpeg 在本机解码 H.264 后重编的 JPEG（质量由 -q:v 决定，非设备端），比设备端 JPEG 多一次解码损失。-rtsp_transport tcp 走 TCP 传 RTP，避免 UDP 丢包花屏。

ONVIF / GB28181

ONVIF 是厂商中立的 SOAP 标准，海康默认关闭，需在 Web 配置页开启并设账号；Python 用 onvif-zeep，GetSnapshotUri() 返回的抓图 URL 仍指向 ISAPI。GB28181 是公安部视频联网国标，基于 SIP+RTP，需配信令服务器。两者价值在跨厂商 / 平台联网，单机取图用不到。

返回 (1440, 2560, 3) uint8，H×W×C，BGR 顺序（OpenCV 惯例，喂 RGB 库前需 cv2.cvtColor）。封装层的价值是把 RTSP 包成 cap.read() 一行取帧、返回 numpy 直接对接下游管线；代价是经解码、且受视频码率限制——存盘体积 ≠ 源质量：opencv_frame.jpg 的 665KB 是 JPEG_QUALITY=95 在主机端撑大的，并非源质量高。恰恰相反，RTSP 帧来自主码流 H.264，CBR 2Mbps / 25fps ≈ 10KB/帧 的压缩预算，解码后用 95 高质量重存只是忠实保留已被压掉细节的画面。故 OpenCV/RTSP 这张体积最大、保真反而最低；而 SDK q0 设备端 JPEG 体积更小、保真更高（相机为单张静帧独立编码，不受视频码率分摊）。

被动推送

相机主动输出图像，适合无人值守或触发式采集：SD 卡本地存储、NAS（NFS/SMB）、FTP 上传（定时或报警）、邮件附件、ISAPI alertStream 长连接事件订阅。

标定取图

标定取图要求单张、全分辨率、像素级无损、参数冻结，与常规监控的连续流取向相反。取图前需关闭相机内部一切会改变像素几何或多次合成的处理：

项目 原因
镜头畸变矫正 / LDC 内部已去畸变，标不出真实畸变
数字防抖 / EIS 裁切、平移画面
宽动态 WDR 多帧合成
降噪 3DNR 时域空域滤波影响亚像素角点
自动曝光 AE 改手动固定快门 / 增益
自动白平衡 AWB 改固定
自动对焦 AF 改手动固定焦距
ROI 裁剪 / 镜像 / 翻转 / 强光抑制 改变像素映射

取主码流 / 主分辨率通道（101 / channel 1）。取图来源上，设备端 JPEG（ISAPI /picture、SDK CaptureJPEGPicture）不经解码，比 RTSP / OpenCV 解码帧更接近原始像素；其中 SDK 可选最高质量档（q0），是标定静态图的首选。内参是相机几何属性、与 JPEG 质量正交——q0 只是给出更精的估计，标定结果可用于任意质量的图像，前提是对焦、焦距、LDC 等几何相关设置在标定与使用时锁成同一状态。
